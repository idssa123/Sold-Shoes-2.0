2) TÍTULO DEL PROYECTO
- Sold Shoes 2.0: Mejora Integral de Tienda Online de roa7pa de Segunda Mano

3) DESCRIPCIÓN DEL PROYECTO
Este anteproyecto propone el desarrollo de una tienda online de ropa de segunda mano con una hoja de ruta por fases. El objetivo es construir una aplicación completa, escalable y usable, cubriendo desde la autenticación de usuarios y gestión de catálogo hasta el proceso de compra y pago.

El alcance se organiza en dos etapas de ejecución:
- Fase 1 (base funcional): autenticación, catálogo, detalle de producto y carrito.
- Fase 2 (mejora integral): pago/checkout, carrito avanzado, ampliación de productos y mejora visual.

La propuesta se centra en cuatro bloques principales:
- Integrar una forma de pago real/simulada para cerrar el proceso de compra.
- Mejorar el carrito para hacerlo más útil y profesional.
- Ampliar notablemente el catálogo de productos.
- Mejorar el diseño visual general y la experiencia de usuario.

Objetivos generales:
- Desarrollar una tienda online completa y cercana a un entorno real de producción.
- Completar el flujo end-to-end: explorar producto -> añadir al carrito -> pagar -> confirmar pedido.
- Mejorar usabilidad, estética y conversión de compra.
- Mantener una arquitectura escalable y mantenible.

Objetivos específicos:
- Incorporar módulo de pago (pasarela externa o simulación académica validada).
- Añadir proceso de checkout con resumen final, dirección y método de pago.
- Mejorar carrito con control de stock, guardado persistente y validaciones.
- Incrementar catálogo con más productos, categorías, filtros y ordenación.
- Rediseñar componentes clave (home, cards, detalle, carrito, checkout) para mejorar visuales.

Requisitos funcionales:
- RF01: El usuario podrá registrarse e iniciar/cerrar sesión.
- RF02: El usuario podrá consultar catálogo con más productos y categorías.
- RF03: El usuario podrá filtrar y ordenar productos (precio, relevancia, novedad).
- RF04: El usuario podrá ver detalle completo del producto.
- RF05: El usuario autenticado podrá añadir productos al carrito y modificar cantidades.
- RF06: El carrito validará stock disponible y recalculará totales automáticamente.
- RF07: El usuario podrá iniciar checkout y seleccionar método de pago.
- RF08: El sistema registrará el resultado del pago (aceptado, rechazado, pendiente).
- RF09: Al confirmar pago, se creará pedido con sus líneas de detalle.
- RF10: El usuario podrá consultar historial/estado de pedidos.

Planificación funcional por fases:
- Fase 1:
  - Registro e inicio de sesión.
  - Catálogo y búsqueda de productos.
  - Detalle de producto.
  - Carrito básico (añadir/eliminar/modificar cantidades).
- Fase 2:
  - Checkout completo con dirección de envío.
  - Integración de pago (simulado o pasarela real).
  - Validación de stock y reglas avanzadas de carrito.
  - Historial de pedidos y estado de pago.
  - Mejora de interfaz y experiencia visual.

Requisitos no funcionales:
- RNF01: Interfaz responsive y visualmente coherente en móvil/tablet/escritorio.
- RNF02: Navegación clara y accesible, con feedback de acciones (carga/error/éxito).
- RNF03: Rendimiento adecuado en catálogo amplio (carga optimizada).
- RNF04: Código modular en componentes y contextos reutilizables.
- RNF05: Persistencia de sesión y de carrito.
- RNF06: Seguridad básica en autenticación y acceso a datos.

4) MODELO ENTIDAD–RELACIÓN (MÍNIMO 4 TABLAS PRINCIPALES)
Nota: Aunque se use Firestore (NoSQL), se presenta el modelo lógico E-R en formato relacional para el diseño académico.

Entidades principales propuestas (8):

A) USUARIOS
- id_usuario (PK)
- nombre
- apellidos
- email (UNIQUE)
- password_hash
- fecha_registro
- estado

B) PRODUCTOS
- id_producto (PK)
- nombre
- descripcion
- precio
- stock
- imagen_url
- categoria
- marca
- talla
- destacado
- fecha_publicacion
- estado

C) CARRITOS
- id_carrito (PK)
- id_usuario (FK -> USUARIOS.id_usuario)
- fecha_creacion
- fecha_actualizacion
- estado_carrito

D) CARRITO_ITEMS
- id_carrito_item (PK)
- id_carrito (FK -> CARRITOS.id_carrito)
- id_producto (FK -> PRODUCTOS.id_producto)
- cantidad
- precio_unitario
- subtotal

E) PEDIDOS
- id_pedido (PK)
- id_usuario (FK -> USUARIOS.id_usuario)
- fecha_pedido
- estado_pedido
- total
- direccion_envio

F) PEDIDO_ITEMS
- id_pedido_item (PK)
- id_pedido (FK -> PEDIDOS.id_pedido)
- id_producto (FK -> PRODUCTOS.id_producto)
- cantidad
- precio_unitario
- subtotal

G) PAGOS
- id_pago (PK)
- id_pedido (FK -> PEDIDOS.id_pedido)
- metodo_pago (tarjeta, bizum, paypal, simulado)
- estado_pago (pendiente, aceptado, rechazado)
- importe
- referencia_pasarela
- fecha_pago

H) CATEGORIAS
- id_categoria (PK)
- nombre
- descripcion
- activa

Relaciones principales:
- USUARIOS 1:N CARRITOS
- CARRITOS 1:N CARRITO_ITEMS
- PRODUCTOS 1:N CARRITO_ITEMS
- USUARIOS 1:N PEDIDOS
- PEDIDOS 1:N PEDIDO_ITEMS
- PRODUCTOS 1:N PEDIDO_ITEMS
- PEDIDOS 1:1 PAGOS (o 1:N si se contemplan reintentos)
- CATEGORIAS 1:N PRODUCTOS

Justificación:
- Se superan ampliamente las 4 tablas principales exigidas.
- Se añade PAGOS para cubrir la nueva necesidad obligatoria del proyecto.
- CATEGORIAS se incluye para soportar más productos y navegación por catálogo.

5) WIREFRAMES DE MEDIA FIDELIDAD (ESTRUCTURA Y NAVEGABILIDAD)
Wireframes propuestos para el desarrollo por fases del proyecto:

WF01 - HOME MEJORADA
Estructura:
- Navbar con accesos claros (Inicio, Productos, Ofertas, Carrito, Perfil).
- Hero con llamada a la acción y beneficios.
- Carrusel visual mejorado de destacados.
- Bloques de categorías y productos recomendados.
Acciones:
- Ir a catálogo/categoría.
- Abrir detalle de producto.

WF02 - CATÁLOGO AVANZADO
Estructura:
- Barra de búsqueda.
- Filtros laterales (categoría, talla, marca, precio).
- Ordenación (precio, más vendidos, novedades).
- Grid de cards rediseñadas.
Acciones:
- Filtrar/ordenar.
- Añadir rápido al carrito.
- Ver detalle.

WF03 - DETALLE DE PRODUCTO
Estructura:
- Galería de imágenes.
- Información completa + stock disponible.
- Selector de talla/cantidad.
- Botón “Añadir al carrito”.
Acciones:
- Seleccionar variante.
- Añadir al carrito.

WF04 - CARRITO MEJORADO
Estructura:
- Lista de ítems con miniatura.
- Controles de cantidad y eliminación.
- Cálculo de subtotal, envío y total.
- Mensajes de validación de stock.
- Botón “Continuar al pago”.
Acciones:
- Editar carrito en tiempo real.
- Ir a checkout.

WF05 - CHECKOUT / PAGO
Estructura:
- Datos de envío.
- Resumen de pedido.
- Selección de método de pago.
- Botón “Confirmar y pagar”.
- Pantalla resultado (éxito/error).
Acciones:
- Completar compra.
- Reintentar pago si falla.

WF06 - PERFIL E HISTORIAL
Estructura:
- Datos de cuenta.
- Lista de pedidos y estado de pago.
Acciones:
- Consultar pedidos anteriores.

Flujo principal actualizado:
HOME -> PRODUCTOS -> DETALLE -> CARRITO -> CHECKOUT/PAGO -> CONFIRMACIÓN
               \-> LOGIN/REGISTRO (si no autenticado)
